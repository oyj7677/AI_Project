# Context

`MyStarNow`의 제품 방향이 "인플루언서 멀티플랫폼 허브"에서 "아이돌 그룹 중심 유튜브 허브"로 바뀌었다. 이 방향에서는 서버가 단순 채널 메타데이터 수집기가 아니라, `그룹-멤버-채널-영상` 관계를 보존하는 조합 계층이어야 한다.

핵심 제약은 아래와 같다.

- 사용자 가치는 채널 단위가 아니라 `그룹 단위 탐색`에서 나온다.
- 공식 채널과 멤버 개인 채널의 관계는 완전 자동 수집만으로 신뢰성 있게 만들기 어렵다.
- 영상 메타데이터 자체는 YouTube 공식 API로 안정적으로 가져올 수 있다.
- 따라서 "무엇을 자동화할지"보다 "어떤 관계를 수동 관리할지"가 더 중요하다.
- 앱과 웹은 동일한 그룹 중심 API 계약을 공유해야 한다.

즉, 이 제품에서 중요한 것은 YouTube 데이터를 많이 가져오는 것이 아니라, **올바른 그룹 구조 안에서 보여주는 것**이다.

# Decision

- 서버 구조는 `Option B: 모듈형 단일 BFF + YouTube 어댑터 워커 + 운영자 관계 관리 데이터`로 결정한다.
- 서버 언어와 프레임워크는 기존 방향을 유지해 `Kotlin + Spring Boot`를 사용한다.
- 플랫폼 범위는 1차에 `YouTube only`로 제한한다.
- 내부 도메인은 `Influencer` 중심이 아니라 `Group`, `Member`, `Channel`, `Video` 중심으로 재설계한다.
- 채널 메타데이터와 영상 목록은 YouTube 공식 API로 자동 수집한다.
- 그룹-멤버-채널 관계와 채널 유형 분류는 운영 입력을 source of truth로 둔다.
- 홈과 상세는 `그룹 중심 읽기 모델`을 사용한다.
- 피드형 응답은 커서 기반 페이지네이션을 기본으로 한다.
- 자동 수집 실패 시 마지막 정상 데이터 + stale/partial 표시 전략을 유지한다.
- 운영 입력 후 읽기 모델은 빠르게 재계산되어야 한다.

# Alternatives Considered

## Option A: YouTube 채널 단위 허브

- 장점: 구조가 단순하다.
- 장점: YouTube 원본 모델과 유사해 구현이 쉽다.
- 단점: 제품 차별점인 `그룹 중심 편의성`이 약해진다.
- 단점: 공식 채널과 멤버 개인 채널을 사용자가 직접 이해해야 한다.

## Option B: 모듈형 단일 BFF + YouTube 어댑터 워커 + 운영자 관계 관리 데이터

- 장점: 그룹 중심 탐색 경험을 서버가 직접 보장할 수 있다.
- 장점: 자동 수집과 수동 관계 보정을 한 읽기 모델로 합칠 수 있다.
- 장점: MVP 단계에서 가장 현실적인 운영 복잡도와 확장성을 가진다.
- 장점: Android/Web 클라이언트가 같은 그룹 중심 API를 재사용할 수 있다.
- 단점: 운영 입력 모델과 읽기 모델 설계를 함께 해야 한다.
- 단점: YouTube 데이터만 충분하지 않고 관계 데이터 품질 관리가 필수다.

## Option C: 완전 자동 관계 추론

- 장점: 운영 입력 비용을 줄일 수 있다.
- 장점: 그룹 수가 매우 많아질 때 이상적으로는 확장성이 좋다.
- 단점: 공식 채널/개인 채널/비공식 채널 오분류 위험이 크다.
- 단점: 초기 제품 신뢰도를 해칠 가능성이 높다.

## Option D: 처음부터 멀티플랫폼 구조 유지

- 장점: 장기적으로는 확장 폭이 넓다.
- 장점: 제품 포지셔닝을 빨리 넓힐 수 있다.
- 단점: 현재 제품 핵심 문제는 YouTube 관계 구조인데, 범위를 넓히면 정확도가 떨어진다.
- 단점: MVP 집중력이 약해진다.

# Consequences

- 긍정적 영향: 그룹 중심 탐색 경험을 데이터 모델 자체로 보장할 수 있다.
- 긍정적 영향: 공식 채널과 멤버 개인 채널의 차이를 명확하게 드러낼 수 있다.
- 긍정적 영향: YouTube API 자동 수집이 가능한 영역과 운영 관리가 필요한 영역이 분리된다.
- 긍정적 영향: 홈 피드와 그룹 상세를 같은 읽기 모델로 단순하게 유지할 수 있다.
- 부정적 영향: 운영자가 그룹/멤버/채널 관계를 잘못 입력하면 서비스 품질이 직접 흔들린다.
- 부정적 영향: 기존 인플루언서 중심 구현과 완전히 같은 데이터 모델을 재사용하기는 어렵다.
- 운영상 주의점: 채널 유형 정의와 운영 가이드를 문서화해야 한다.
- 운영상 주의점: 자동 수집값보다 운영 보정값의 우선순위를 명확히 해야 한다.
- 운영상 주의점: 홈 피드와 그룹 상세 피드의 페이지네이션 기준을 일관되게 유지해야 한다.

# Follow-up Decisions

- 그룹/멤버/채널/영상 테이블 설계를 별도 DB schema 문서로 확정한다.
- 그룹 목록, 그룹 상세, 멤버 상세 API 계약을 OpenAPI 문서로 별도 작성한다.
- YouTube sync scope를 `channel_profile`, `channel_videos` 두 축으로 재정의한다.
- 운영자 입력 경로에 `group`, `member`, `channel`, `video` 생성 API를 추가한다.
- 기존 인플루언서 중심 화면과 새 그룹 중심 화면의 마이그레이션 전략을 정한다.

# Rollout / Migration

- 1단계: 새 제품 방향용 PRD/TRD/ADR를 기준 문서로 확정한다.
- 2단계: 서버 DB 모델을 그룹 중심 구조로 확장한다.
- 3단계: 운영자 입력 경로를 그룹/멤버 기준으로 추가한다.
- 4단계: YouTube 자동 수집을 그룹 중심 읽기 모델에 연결한다.
- 5단계: 홈/그룹 상세/멤버 상세 API를 순차적으로 구현한다.
