# Summary

`MyStarNow Idol YouTube BFF`는 아이돌 그룹별 공식 유튜브 채널과 멤버 개인 유튜브 채널을 수집·정규화·조합해 그룹 중심 탐색 경험을 제공하는 백엔드 제품이다. 서버는 단순 영상 목록 API가 아니라 `그룹-멤버-채널-영상` 관계를 관리하고, Android/Web 클라이언트가 그룹 중심 화면을 바로 구성할 수 있는 응답을 제공해야 한다. 1차 범위는 `YouTube only`이며, 관계 데이터는 운영자가 관리하고 채널/영상 메타데이터는 공식 API 기반으로 자동 수집한다.

# Problem

- 아이돌 그룹의 공식 유튜브와 멤버 개인 유튜브가 분리돼 있어 팬이 여러 채널을 직접 찾아가야 한다.
- 사용자가 궁금한 것은 "이 그룹이 최근에 유튜브에서 뭘 올렸는가"인데, YouTube 원본 구조는 그룹 중심이 아니라 채널 중심이다.
- 공식 채널, 멤버 개인 채널, 서브 유닛 채널, 레이블 운영 채널의 관계는 자동 수집만으로 정확히 알기 어렵다.
- 신규 팬은 그룹 입문 시 어떤 채널이 공식이고 어떤 채널이 멤버 개인 채널인지 파악하기 어렵다.
- 앱이 YouTube API를 직접 붙이면 채널 관계 관리, 화면 조합, fallback 처리가 클라이언트에 분산된다.

# Goals

- 클라이언트가 `그룹 중심`으로 탐색할 수 있도록 단일 BFF API를 제공한다.
- 그룹별 공식 채널과 멤버 개인 채널을 명확하게 구분해 제공한다.
- 그룹 통합 최근 업로드 피드를 최신순으로 제공한다.
- 그룹, 멤버, 채널 관계는 운영자가 빠르게 입력/수정할 수 있게 한다.
- 채널 메타데이터와 최신 영상 목록은 YouTube 공식 API로 자동 수집한다.
- 홈/목록/상세 화면에 필요한 데이터가 추가 조합 없이 바로 사용 가능한 형태로 제공되도록 한다.
- 1차 제품 검증에 필요한 데이터 정확도와 운영 속도를 확보한다.

# Non-Goals

- 1차 범위에서 Instagram, X, TikTok, Weverse 등 타 플랫폼을 연동하지 않는다.
- 1차 범위에서 자체 영상 재생, 댓글, 구독, 알림 기능을 제공하지 않는다.
- 1차 범위에서 YouTube 외 미디어 스케줄, 음원, 굿즈, 공연 정보까지 관리하지 않는다.
- 1차 범위에서 자동으로 그룹-멤버-채널 관계를 추론하는 완전 자동화 시스템을 만들지 않는다.
- 1차 범위에서 범용 대외 공개 Open API를 만들지 않는다.
- 1차 범위에서 대규모 관리자 CMS를 만들지 않는다.

# Users / Scenarios

## Primary Users

- `MyStarNow` 클라이언트 앱
- 그룹/멤버/채널 관계를 관리하는 운영자

## Core Scenarios

- 앱 홈이 여러 그룹의 최근 유튜브 업로드를 그룹 중심 피드로 조회한다.
- 앱 그룹 목록이 그룹명, 최근 활동 시각, 공식 채널 수, 멤버 개인 채널 수를 조회한다.
- 앱 그룹 상세가 공식 채널 섹션, 멤버 섹션, 그룹 통합 영상 피드를 한 번에 조회한다.
- 앱 멤버 상세가 특정 멤버의 개인 채널과 최근 영상을 조회한다.
- 운영자가 새 그룹, 새 멤버, 새 채널 관계를 등록한다.
- 운영자가 공식 채널/개인 채널 분류 오류를 수정한다.
- 운영자가 자동 수집 전이라도 수동 영상 등록으로 피드를 보정한다.

# Scope

## Included

- 그룹 메타데이터 저장
- 멤버 메타데이터 저장
- 그룹-멤버 관계 저장
- 그룹/멤버 소유 채널 정보 저장
- 채널 유형 관리
  - `GROUP_OFFICIAL`
  - `MEMBER_PERSONAL`
  - `SUB_UNIT`
  - `LABEL`
- YouTube 채널 메타데이터 자동 수집
- YouTube 최신 영상 목록 자동 수집
- 그룹 중심 홈 피드
- 그룹 목록 조회
- 그룹 상세 조회
- 멤버 상세 조회
- 그룹/채널/영상 운영자 입력 경로
- 캐시, stale, degraded mode 대응
- 관측성 로그와 운영 지표

## Excluded

- Instagram 등 타 플랫폼 수집
- 라이브 스트리밍 재생 기능
- 팬 커뮤니티 기능
- 개인화 추천
- 푸시 알림 발송
- 대규모 공개 검색 엔진 최적화 기능

# Server Responsibilities

## Group-centric BFF API

- 그룹, 멤버, 채널, 영상 정보를 그룹 중심 구조로 정규화해 응답한다.
- 클라이언트가 YouTube 원본 API 형태를 몰라도 화면을 구성할 수 있게 한다.

## Relationship Management

- 그룹과 멤버의 연결
- 멤버와 개인 채널의 연결
- 그룹과 공식 채널의 연결
- 채널 유형 및 공식 여부 관리

## Content Aggregation

- YouTube 채널별 영상 목록을 수집한다.
- 그룹 상세와 홈 피드를 위해 여러 채널의 영상을 그룹 기준으로 재정렬한다.
- 그룹 피드에서 채널 유형과 멤버/그룹 소유 정보를 함께 내려준다.

## Operator-managed Correction

- 관계 데이터와 메타데이터는 운영 입력을 통해 보정한다.
- 자동 수집 전이나 API 실패 시 수동 영상 등록으로 제품 화면을 유지한다.

# Success Metrics

- 그룹 상세 API 응답 p95 700ms 이내
- 홈 그룹 피드 API 응답 p95 500ms 이내
- 그룹 신규 등록 후 10분 이내 화면 노출 가능
- 운영자가 신규 그룹 온보딩을 10분 이내 완료 가능
- 최근 영상 자동 수집 반영 시간 30분 이내
- 관계 데이터 수정 후 5분 이내 읽기 모델 반영

# Acceptance Criteria

- [ ] 서버가 그룹, 멤버, 채널, 영상을 그룹 중심 읽기 모델로 제공한다.
- [ ] 클라이언트는 단일 BFF API만으로 그룹 목록/상세/멤버 상세를 구성할 수 있다.
- [ ] 운영자는 그룹, 멤버, 채널 관계를 등록 및 수정할 수 있다.
- [ ] 서버는 그룹 통합 최근 업로드 피드를 최신순과 더보기 형태로 제공한다.
- [ ] 채널은 공식 채널과 멤버 개인 채널을 구분해 응답한다.
- [ ] YouTube 공식 API 기반 자동 수집 경로가 존재한다.
- [ ] 자동 수집 실패 시 마지막 정상 데이터 또는 수동 데이터로 fallback 한다.

# Risks

- 그룹-멤버-채널 관계가 복잡한 경우 운영 입력 비용이 커질 수 있다.
- 공식 채널과 비공식 채널을 잘못 연결하면 신뢰도가 크게 떨어진다.
- 채널 유형 정의가 모호하면 UI와 데이터 모두 흔들릴 수 있다.
- YouTube quota 설계를 잘못하면 그룹 수가 늘어날수록 수집 안정성이 떨어질 수 있다.
- 멤버 개인 채널 여부를 판단하는 운영 기준이 일관되지 않으면 품질 관리가 어려워진다.

# Open Questions

- 멤버 상세를 1차 출시 범위에 포함할지
- 그룹 대표 이미지와 멤버 이미지의 source of truth를 어디로 둘지
- 서브 유닛 채널을 그룹 공식 채널과 같은 수준으로 노출할지
- 홈에서 "최근 업로드 피드"와 "그룹 추천"을 함께 둘지, 피드 중심으로 단순화할지
- 그룹별 즐겨찾기 외에 멤버 단위 즐겨찾기도 필요한지
